# Спецификация MVP многофункционального чат-бота

**Проект:** «Разработка многофункционального чат-бота с интеграцией GigaChat и ProxyAPI» (заказ с фриланс-маркетплейса)
**Стек:** GigaChat API (через ProxyAPI), Telegram, Python
**Бюджет:** 7 000 (по условиям площадки) · **Дедлайн:** 24 июля 2026
**Дата спецификации:** 17 июля 2026

---

## Заказчик и задача

Заказ на разработку многофункционального Telegram-бота для интернет-магазина/маркетплейса. Задача бота — информировать покупателей о товарах, проверять статус заказа и отвечать на вопросы поддержки. Реальной CRM и системы заказов у заказчика нет — используются тестовые заглушки, которые в будущем можно заменить на настоящие интеграции. Готового FAQ/прайса заказчик не предоставил — сценарии и база знаний разрабатываются самостоятельно на основе заданных тем, с упором на простоту и понятность для конечного клиента.

## Основные задачи бота

1. Отвечать на вопросы о товарах/продуктах на основе демо-каталога.
2. Проверять статус заказа по номеру (через тестовую заглушку).
3. Отвечать на вопросы технической поддержки.
4. При нехватке данных или сложном вопросе — переводить диалог на менеджера, не придумывая ответ.
5. Собирать контакт только с явного согласия клиента.
6. Логировать статистику: число диалогов, время ответа, типы вопросов (без учёта конверсии — она не требуется).

## Типовые вопросы клиентов (5–8)

1. Какие товары/услуги у вас есть, расскажите о продукте?
2. Какой статус у моего заказа №___?
3. Когда ожидать доставку заказа?
4. Как отменить или изменить заказ?
5. Что делать, если товар пришёл повреждённым или не тот?
6. Как связаться с поддержкой, если бот не смог помочь?
7. Какие способы оплаты доступны?
8. Есть ли гарантия / как оформить возврат?

## Целевая аудитория

B2C — обычные покупатели интернет-магазина/маркетплейса, без технической подготовки. Формулировки заказчика ("просто и понятно для клиентов") прямо указывают на этот сегмент. Тон бота: дружелюбный, простой язык, без жаргона, короткие и понятные ответы.

## Платформа и причина выбора

**Telegram.** Заказчик оставил выбор платформы на исполнителя; группы/канала нет — создаётся с нуля. Выбор в пользу Telegram обоснован:
- Срок в 7 дней — критично использовать уже проверенный стек (aiogram, gigachat/httpx, tenacity) из предыдущего проекта, а не осваивать VK API с нуля.
- Telegram Bot API проще интегрируется с FSM для нескольких сценариев (товары / статус заказа / поддержка) в сжатые сроки.

## Интеграции

| Компонент | Решение | Обоснование |
|---|---|---|
| Нейросеть | GigaChat через ProxyAPI (`gigachat` SDK или `httpx`) | Управление авторизацией, retry, async — знакомо из прошлого проекта |
| Платформа | Telegram Bot API через `aiogram` | Переиспользование готовой архитектуры, быстрая разработка |
| CRM / обращения | Тестовая заглушка REST API (`POST /leads`) | Реальной системы нет — готова к замене |
| Статус заказа | Тестовая заглушка REST API (`GET /orders/{order_id}`) | Новый компонент по сравнению с прошлым проектом — имитирует систему заказов |
| Хранилище статистики | SQLite или файл (JSON/CSV) | Заказчик не просматривает отчёты регулярно — достаточно простого лога, доступного для разбора при необходимости |

**Структура заглушки статуса заказа:**
```json
{
  "order_id": "строка",
  "status": "в обработке | собран | в доставке | доставлен | отменён",
  "estimated_delivery": "ISO-date",
  "last_updated": "ISO-timestamp"
}
```

**Структура заглушки обращения (лида/тикета поддержки):**
```json
{
  "ticket_id": "uuid",
  "contact": "telegram_id (только с согласия)",
  "topic": "товары | статус заказа | поддержка",
  "message": "текст обращения",
  "status": "new | needs_manager",
  "created_at": "ISO-timestamp"
}
```

## База знаний для бота (товары, статус заказа, FAQ)

Готового прайса/FAQ нет — создаётся демо-база с явно помеченными тестовыми значениями:
- Каталог товаров (заглушка — 3–5 демо-позиций с названием, кратким описанием, ценой)
- Статусы заказов (заглушка — тестовые order_id с разными статусами для проверки сценариев)
- FAQ по поддержке (заглушка — типовые ответы по возврату, оплате, доставке)

Все значения должны быть явно закомментированы в коде как "ЗАГЛУШКА — заменить на реальные данные заказчика".

## Ограничения и правила поведения

- Не выдумывать факты, цены, сроки доставки и гарантии сверх того, что есть в базе знаний/заглушке заказа.
- При отсутствии данных (например, order_id не найден) — не угадывать статус, а сообщить об этом и предложить связаться с менеджером.
- Собирать контактные данные пользователя только после явного согласия — без согласия не запрашивать и не сохранять.
- Ключи (GigaChat/ProxyAPI, Telegram Bot Token) хранятся в переменных окружения (`.env`), не передаются в промпт нейросети и не логируются.
- Тон общения — простой, дружелюбный, без сложных терминов (в отличие от B2B-проекта).
- Ограничение темы диалога рамками магазина/заказов/поддержки.

## Обработка ошибок и нестабильности

| Тип сбоя | Обработка |
|---|---|
| Сетевые сбои/таймауты GigaChat/ProxyAPI | Таймаут запроса (15–20 сек), retry с экспоненциальной задержкой (`tenacity`, 2–3 попытки) |
| Лимиты/авторизация (429, протухший токен) | Автообновление токена, пауза и повтор при превышении лимита |
| order_id не найден в заглушке | Явное сообщение "не нашла заказ с таким номером, уточните номер или свяжитесь с менеджером" — без выдуманного статуса |
| Невалидный/пустой ответ модели | Валидация перед отправкой пользователю, заготовленная безопасная фраза при сбое |
| Полная недоступность нейросети | Fallback-сообщение + передача диалога менеджеру, логирование инцидента |
| Сбои платформы (Telegram API) | Очередь отправки сообщений с повтором |

## Промпт для нейросети-разработчика (ClaudeCode)

```
Разработай MVP многофункционального Telegram-бота для интернет-магазина на Python.

СТЕК:
- aiogram (последняя стабильная версия) — Telegram Bot API, async, FSM для сценариев
- gigachat SDK или httpx — вызовы к GigaChat через ProxyAPI
- pydantic — валидация запросов/ответов и конфигурации
- python-dotenv — хранение ключей (GIGACHAT_CREDENTIALS, PROXYAPI_KEY, TELEGRAM_BOT_TOKEN) в .env
- tenacity — retry с экспоненциальной задержкой при сетевых сбоях/лимитах (2-3 попытки)
- sqlite3 (stdlib) — хранение статистики (число диалогов, время ответа, тип вопроса)
- logging (stdlib) — логирование ошибок и диалогов

СТРУКТУРА ПРОЕКТА:
/bot
  main.py             — точка входа, инициализация aiogram
  handlers.py         — обработчики сообщений, команд и FSM-сценариев (товары/статус заказа/поддержка)
  gigachat_client.py  — обёртка над GigaChat/ProxyAPI с retry и валидацией ответа
  knowledge_base.py   — демо-каталог товаров и FAQ поддержки — все значения помечены как ТЕСТОВЫЕ/ЗАГЛУШКА
  order_stub.py       — тестовая заглушка REST API для проверки статуса заказа (GET /orders/{order_id})
  ticket_stub.py      — тестовая заглушка REST API для обращений/лидов (POST /tickets)
  stats.py            — запись статистики (тип вопроса, время ответа) в SQLite
  config.py           — загрузка .env, Pydantic-настройки
.env.example
requirements.txt

ФУНКЦИОНАЛЬНОСТЬ:
1. При старте диалога — приветствие простым языком, короткое меню/подсказка: товары, статус заказа, поддержка.
2. Вопросы о товарах: поиск в knowledge_base.py → ответ через GigaChat в рамках переданного контекста.
3. Проверка статуса заказа: запрос номера заказа у пользователя → вызов order_stub.py → если найден, вернуть статус и дату доставки; если не найден — сообщить об этом и предложить связаться с менеджером, НЕ придумывать статус.
4. Вопросы поддержки: ответ на основе FAQ из knowledge_base.py; если ответа нет — переводить на менеджера.
5. Сбор контакта — только после явного согласия пользователя ("Хотите, чтобы менеджер связался с вами?"). Без согласия контакт не запрашивается и не сохраняется.
6. При эскалации к менеджеру — создание тикета через ticket_stub.py.
7. Логирование каждого диалога: тип вопроса (товары/заказ/поддержка), время ответа — запись в stats.py. Конверсия и регулярные отчёты не требуются.
8. Обработка ошибок API (таймауты, 429, невалидный ответ) — через retry, при исчерпании попыток — fallback-сообщение + передача менеджеру, без технических деталей пользователю.

ОГРАНИЧЕНИЯ (обязательно закрепить в системном промпте нейросети):
- Не придумывать цены, сроки доставки, статусы заказов и гарантии, отсутствующие в базе знаний/заглушке заказа.
- При неуверенности — явно предлагать связаться с менеджером, а не гадать.
- Никогда не включать в промпт нейросети содержимое .env/ключей API.
- Запрашивать и сохранять контактные данные пользователя только после явного согласия.
- Отвечать простым, дружелюбным языком без сложных терминов.

Добавь unit-тесты для order_stub.py, ticket_stub.py и gigachat_client.py (мок ответов). Все тестовые/демо-данные должны быть явно закомментированы как "ЗАГЛУШКА — заменить на реальные данные заказчика".
```

## Два варианта поведения при нехватке данных

**Слабый вариант (не делать так):** бот сам придумывает статус заказа — например, "ваш заказ уже отправлен и придёт завтра", хотя номер заказа не найден в системе или данных о нём нет. Это создаёт ложные ожидания у клиента и риск конфликта с реальным статусом заказа.

**Правильный вариант:** бот честно сообщает о границе своих знаний — "Не нашла информацию по заказу №___, чтобы не сообщить вам неверный статус, я передам это менеджеру — он уточнит и свяжется с вами" — и создаёт тикет для менеджера, чтобы обращение не потерялось.
