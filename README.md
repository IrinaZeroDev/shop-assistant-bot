# Многофункциональный чат-бот интернет-магазина (MVP)

Telegram-бот для интернет-магазина/маркетплейса: отвечает на вопросы о
товарах, проверяет статус заказа по номеру, отвечает на вопросы поддержки
(доставка, оплата, возврат, гарантия). Полная спецификация — в файле
`Спецификация MVP чат-бота (многофункциональный).md`, исходное ТЗ — в `ТЗ.txt`.

**Важно:** заказчик явно подтвердил, что реальных данных (каталога, CRM,
системы заказов) не будет никогда — все интеграции остаются тестовыми
заглушками постоянно, это не временное решение до подключения боевых систем.

## Стек

Python 3.11+, aiogram 3 (Telegram), httpx + tenacity (GigaChat через
ProxyAPI, retry с экспоненциальной задержкой), pydantic / pydantic-settings,
SQLite (статистика и тикеты).

## Структура проекта

```
bot/
  config.py           — настройки из .env
  models.py            — Ticket, OrderStatus, DialogSession и связанные модели
  knowledge_base.py    — демо-каталог товаров и FAQ поддержки (ЗАГЛУШКА — постоянная)
  gigachat_client.py   — обёртка над GigaChat/ProxyAPI: retry, валидация, mock-режим
  order_stub.py         — заглушка проверки статуса заказа (GET /orders/{order_id})
  ticket_stub.py         — заглушка приёма обращений (POST /tickets), SQLite
  stats.py                — статистика диалогов в SQLite (без конверсии)
  messages.py             — канн-реплики, системный промпт, ключевые слова
  dialog.py                — движок диалога (не зависит от Telegram)
  rate_limit.py             — антиспам-троттлинг (aiogram middleware)
  handlers.py                — aiogram-обработчики (тонкий транспортный слой)
  main.py                     — точка входа Telegram-бота
console_test.py               — консольный тест-харнесс (без Telegram и без ключей)
tests/                         — unit-тесты
```

## Установка

```bash
python -m venv .venv
.venv\Scripts\activate        # Windows
pip install -r requirements-dev.txt
```

## Тестирование без ключей (доступно уже сейчас)

Без `PROXYAPI_KEY` бот автоматически переходит в офлайн-режим
(`MockGigaChatClient`) и отвечает напрямую на основе базы знаний — всю
логику диалога можно проверить локально, без сети.

```bash
python console_test.py
```

Тестовые номера заказов для сценария «статус заказа»: `12345`, `67890`,
`11111`, `22222`, `99999` — любой другой номер считается ненайденным (как и
будет с реальным несуществующим order_id).

Пример сценария для проверки:

1. `Какие товары у вас есть?` (или команда `каталог`) — бот покажет демо-каталог.
2. `статус заказа` → `12345` — бот вернёт статус и дату доставки.
3. `статус заказа` → `00000` — бот честно сообщит, что не нашёл заказ, и создаст тикет.
4. `Товар пришёл повреждённым` — бот ответит по FAQ и предложит связать с менеджером.
5. `да` — тикет создаётся, диалог продолжается.
6. `пока` → диалог завершается.

Проверить, что записалось:

```bash
python console_test.py --show-tickets
python console_test.py --show-stats
```

## Unit-тесты

```bash
pytest
```

Тесты мокают сетевые вызовы (GigaChat/ProxyAPI) через `unittest.mock`,
реальная сеть не используется.

## Как подключён GigaChat (через ProxyAPI)

Архитектура идентична первому проекту (`sale_chart-bot`): `dialog.py`
классифицирует тему по ключевым словам → берёт контекст из
`knowledge_base.py` → формирует системный промпт, ограничивающий модель
отвечать только на основе контекста → `gigachat_client.py` вызывает ProxyAPI
с retry. Без `PROXYAPI_KEY` используется офлайн-заглушка. Подробности и что
нужно для подключения боевого ключа — см. README первого проекта, раздел
«Как подключён GigaChat» (логика не отличается).

## Ограничения и правила поведения (реализовано)

- Бот отвечает только на основе базы знаний/заглушки заказа — не придумывает
  цены, сроки доставки, статусы заказов и гарантии.
- Контактные данные (telegram_id) используются только после явного согласия
  клиента, кроме автоматической эскалации при "заказ не найден" — это не сбор
  личных данных, а фиксация самого обращения, чтобы оно не потерялось.
- Если заказ не найден — бот честно сообщает об этом, а не угадывает статус.
- Вопросы не по теме магазина/заказов/поддержки вежливо ограничиваются
  рамками сценария, без обращения к нейросети.
- Ключи хранятся только в `.env`, не логируются и не передаются в промпт.

## Известные ограничения MVP

- Сессии диалога хранятся в памяти процесса (`bot/handlers.py`) — при
  перезапуске бота диалог с клиентом начинается заново. Статистика и тикеты
  сохраняются в SQLite и не теряются.
- Классификация темы вопроса — rule-based по ключевым словам
  (`bot/knowledge_base.py: find_topic`), без ML-слоя — достаточно для 8
  типовых тем MVP.
- GigaChat пока не подключён боевым ключом — работает в mock-режиме на
  базе знаний.
- Каталог товаров, статусы заказов и FAQ — постоянные тестовые заглушки по
  явному подтверждению заказчика (не временное решение).
